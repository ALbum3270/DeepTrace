## 项目概览（Overview）

### 项目名称（暂定）

事件链侦探系统 / EventChain Investigator  
（你后面可以改成你喜欢的名字）

### 背景

传统舆情系统的典型模式是：

- 先“海量爬取”与某个话题相关的内容（帖子 + 评论）
- 再在大数据上做统计、情感分析、词云、趋势图
- 最后给出一份“宏观报告”

这种模式存在几个痛点：

- **信息过载**：抓了一堆数据，但对“事件真实来龙去脉”帮助有限
- **无法按需深挖**：很难像人类侦探一样，围绕一个关键线索逐步深入
- **效率低 / 成本高**：对反爬强的平台（如小红书）很难做“全量爬取”
- **评论被浪费**：评论里常藏着关键证据、补充事实和不同版本，但往往只被当成“情感样本”

本项目目标是做一个 **“思维链式 / 侦探式舆情深挖系统”**，优先支持小红书等中文内容平台，重点在于：

> 围绕“某个具体事件”，按需搜索 & 按需爬取，在有限的数据预算下，还原尽可能完整的 **事件起因–经过–关键节点–结果**。

### 核心目标

1. **事件链深挖，而不是简单统计**
   - 给定一个事件问题（例如：“XX 品牌在小红书上的翻车到底怎么回事？”）
   - 系统自动构建并迭代一条 **事件因果链**：
     - 起因 → 诱因 → 事件爆发 → 多方反应 → 关键转折 → 结果 / 当前状态

2. **按需检索 & 按需爬取，而非先全量抓**
   - 通过 LLM + Agent 规划：
     - 当前时间线缺什么信息？
     - 哪个平台 / 哪个时间段 / 哪类内容值得再查？
   - 每次只做 **小批量 / 高价值** 的检索或爬取，控制成本与风控压力

3. **评论驱动的“二次深挖”**
   - 在初次抓到的帖子下，对评论进行：
     - 新信息 / 新实体识别
     - 证据型内容识别（截图、链接、具体细节）
     - 与现有叙事的冲突点识别
   - 对“关键评论”自动生成子问题，推动二轮检索 & 证据补全

4. **可审计、可解释**
   - 最终生成的事件链中，每个节点都需要：
     - 有对应的证据列表（帖子 / 评论 / 新闻链接等）
     - 标注证据来源、时间、置信度
   - 支持“点事件节点 → 展开原始证据”

5. **支持多平台，但不追求一上来就“全网”**
   - 优先支持：
     - 小红书（核心实验场）
     - 微博 / 新闻网站（配合作为外部佐证）
   - 后续再考虑：知乎、B 站、短视频平台等

### 非目标（当前阶段不做）

- 不做真正的平台级 **传播路径图 / 转发树**（这需要平台内部行为日志）
- 不做 7×24 全网实时监控，只做 **按需深挖型分析工具**
- 不做完整商业化舆情系统（账号管理、客户报表等），主要面向：
  - 研究者 / 分析师 / 个人爱好者
- 不绕过任何平台的安全机制或风控限制，遵守各平台服务条款与当地法律

### 目标用户 & 使用场景（简要）

- **品牌方 / PR / 市场分析师**
  - 想要搞清楚某次“翻车”背后的真实因果链，而不是只看情感饼图
- **研究者 / 媒体 / 自媒体**
  - 想快速梳理一个舆论事件的发展过程及关键节点
- **个人玩家 / 技术爱好者**
  - 想体验“让 AI 当舆情侦探”的工作方式

### 高层系统能力

- 事件查询接口（输入自然语言问题）
- 证据存储层（Evidence Store）
- 事件节点抽取 & 时间线构建
- 评论线索挖掘 & 关键评论识别
- 检索 / 爬取规划（Retrieval Planner）
- 目标内容抓取器（可对接小红书 / 微博 / 新闻检索等）
- 事件链迭代引擎（不断补证 → 更新链条）
- 报告生成器（文本 + 可视化 / 节点列表）
