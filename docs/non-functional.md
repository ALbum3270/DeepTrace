## 非功能需求 & 边界条件（Non-functional & Boundaries）

### 一、数据 & 合规边界

1. **只使用公开可访问数据**
   - 不尝试绕过登录 / 加密 / 反爬虫机制
   - 不抓取非公开页面 / 私域内容（例如私信、非公开群）

2. **平台合规**
   - 遵守各平台的服务条款与 robots 约束
   - 访问频率限制在“人类正常使用”范围内，避免造成服务压力

3. **用途边界**
   - 仅限学习、研究、自用分析
   - 不直接对外提供商业舆情监控服务（除非后续单独合规评估）

### 二、性能 & 规模

1. **目标规模（MVP 阶段）**
   - 面向“单事件深挖”场景：
     - 证据条目：1k–10k 级别（帖子 + 评论）
     - 事件节点：几十个级别
   - 不追求覆盖百万级全网数据

2. **响应时间**
   - 事件链初稿：在种子检索完成后的几分钟内生成
   - 一次迭代（按需补证）：
     - 检索 / 爬取：取决于外部接口，但需有超时与重试控制
     - 事件链更新：在几十秒内完成（不含外部 I/O）

3. **资源使用**
   - 设计成可以在单机（有 GPU 更好，但非必需）上运行
   - 大模型调用可对接外部 API 或本地部署模型，需可配置

### 三、可靠性 & 健壮性

1. **错误处理**
   - 外部接口失败（超时 / 429 / 反爬）：
     - 记录错误类型
     - 尝试退避重试
     - 不因为单次失败导致整条管线崩溃
   - LLM 返回不符合预期格式时：
     - 有解析兜底 & 日志记录
     - 能给出明确的失败说明

2. **可重复性**
   - 对同一查询，在同一配置下，多次运行结果应在“同一大框架”内小幅波动，而不是完全不同的故事
   - 支持保存：
     - 原始证据快照
     - 事件链版本（v1, v2, v3…）

### 四、可解释性 & 透明度

1. **证据追溯**
   - 每个事件节点必须关联：
     - 证据 ID 列表
     - 每条证据的来源、时间、平台
   - 报告中关键结论必须可点击 / 展开查看对应证据摘要

2. **不确定性表达**
   - 对证据不足 / 存在冲突的结论：
     - 标注为“中等 / 低置信度”
     - 给出“其他可能解释”的摘要，而不是装作“唯一真相”

### 五、系统边界 & 不做的事

1. **不做完整用户管理 / 权限系统**
   - MVP 阶段只假设单用户 / 小团队使用
   - 安全需求主要是本机访问控制 & 配置文件安全

2. **不做复杂可视化前端**
   - 初期通过：
     - Markdown / HTML 报告
     - 简单的节点列表 & 时间线图
   - 复杂交互（拖拽图谱、动态图）可以后续迭代

3. **不承诺还原“绝对真相”**
   - 目标是“在公开信息范围内给出 **尽可能合理的因果链**”
   - 明确提示：仍有不可见信息 / 封闭决策 / 删除内容等不可控因素
